# Specification: Scaffold Backend

> Greenfield backend scaffold for BarberPro. 8 new capabilities, zero existing specs.

---

## 1. Project Setup & Configuration

**Description**: UV-managed Python project with DDD directory structure, Pydantic Settings, `.env.example`, and `.gitignore`.

**Acceptance Criteria**:
- `uv sync` installs all deps without errors
- `uv run python -c "from app.core.config import settings"` loads config from env
- All 8 capability directories exist under `app/` and `tests/`
- `.env.example` documents every env var with a placeholder
- `.gitignore` excludes `__pycache__/`, `.env`, `*.pyc`, `.venv/`, `.DS_Store`

**Contracts**:

```python
# app/core/config.py
class Settings(BaseSettings):
    DATABASE_URL: str  # postgresql+asyncpg://user:pass@host:5432/barberpro
    REDIS_URL: str     # redis://default:pass@localhost:6379/0
    SECRET_KEY: str
    JWT_ALGORITHM: str = "HS256"
    JWT_EXPIRY_HOURS: int = 24
    N8N_WEBHOOK_URL: str = ""
    CORS_ORIGINS: list[str] = ["http://localhost:5173"]
    ENVIRONMENT: str = "development"
```

**Dependencies**: UV ≥0.4, Python ≥3.12

## 2. Docker Infrastructure

**Description**: `docker-compose.yml` with PostgreSQL 16 + Redis 7, health checks, named volumes, internal bridge network.

**Acceptance Criteria**:
- `docker compose up -d` starts both containers healthy within 30s
- PostgreSQL accessible on port 5432 with configured db/user/password
- Redis accessible on port 6379 with configured password
- Named volumes survive `docker compose down`
- Containers share an internal network `barberpro-net`
- Health checks prevent app startup if DB/Redis not ready

**Scenarios**:

- GIVEN Docker is installed AND `docker compose up -d` is run
  WHEN all containers report `healthy`
  THEN `psql -h localhost -U barberpro -d barberpro -c '\l'` succeeds
  AND `redis-cli -a <password> ping` returns `PONG`

- GIVEN `docker compose down` is executed
  WHEN `docker compose up -d` is run again
  THEN PostgreSQL data from the previous session is preserved

**Contracts**: `docker-compose.yml` — services, volumes, networks. No Pydantic schemas.

**Dependencies**: Docker Engine ≥24, Docker Compose v2

## 3. Database & Migrations

**Description**: SQLAlchemy async engine + session factory, Alembic async migrations via `run_async()`, Base model with common columns.

**Acceptance Criteria**:
- `alembic upgrade head` creates all tables without errors using async env.py
- Engine is `create_async_engine(async_db_url)` with echo configurable
- `async_sessionmaker` produces async sessions with `expire_on_commit=False`
- `Base` model class has `id: UUID` (PK), `created_at: datetime`, `updated_at: datetime`
- Alembic env.py uses `asyncio.run(run_async())` at top level, `run_async()` calls `run_migrations()`
- Migration is autogenerated: `alembic revision --autogenerate -m "initial"`

**Scenarios**:

- GIVEN the database is empty AND `alembic upgrade head` is run
  WHEN migration completes
  THEN all entity tables exist in the database
  AND the `alembic_version` table records the migration revision

- GIVEN Alembic env.py is configured
  WHEN `alembic upgrade head` runs
  THEN the engine is created and disposed within the async context
  AND no "coroutine was never awaited" warnings appear

**Contracts**: No Pydantic schemas. Alembic configuration in `alembic.ini` + `alembic/env.py`.

**Dependencies**: PostgreSQL 16 running (Docker), SQLAlchemy 2.x, Alembic 1.x

## 4. Domain Entities

**Description**: Six pure-Python entities (dataclasses/plain classes) in `app/domain/entities/`. No framework imports, no ORM coupling.

**Acceptance Criteria**:
- Each entity is a plain class or `@dataclass` with type-annotated fields
- No imports from `sqlalchemy`, `fastapi`, or any framework
- Entity validation uses Python built-ins (`ValueError`, `@property`) — no Pydantic
- All six entities exist and match the model below

**Contracts**:

```python
# Domain entities (plain classes, no framework imports)

class Barber:
    id: UUID | None
    name: str
    phone: str
    price: Decimal | None          # optional override per barber

class Service:
    id: UUID | None
    name: str
    price: Decimal
    duration_minutes: int

class BarberService:
    barber_id: UUID
    service_id: UUID
    price: Decimal | None           # override barber's service price
    duration_minutes: int | None    # override barber's service duration

class Availability:
    id: UUID | None
    barber_id: UUID
    day_of_week: int                # 0=Monday .. 6=Sunday
    start_time: time
    end_time: time

class Appointment:
    id: UUID | None
    date: date
    time: time
    barber_id: UUID
    service_id: UUID
    client_name: str

class Admin:
    id: UUID | None
    username: str
    hashed_password: str
```

**Scenarios**:

- GIVEN a `Barber` entity is created with `name="Carlos"`, `phone="+5491112345678"`
  WHEN accessing `.name`
  THEN it returns `"Carlos"`

- GIVEN an `Appointment` is created with `status="invalid"`
  WHEN the entity validates
  THEN a `ValueError` is raised with a message listing valid statuses

- GIVEN a `Service` is created with `price=Decimal("0.00")`
  WHEN the entity validates
  THEN a `ValueError` is raised (price MUST be positive)

**Dependencies**: None — pure Python standard library only

## 5. API Endpoints

**Description**: FastAPI routes under `/api/v1/` — public booking, admin CRUD, auth.

**Acceptance Criteria**:
- All endpoints return correct HTTP status codes (200, 201, 4xx)
- Protected endpoints return 401/403 without valid auth
- Request bodies validated via Pydantic schemas
- Response bodies match specified schemas
- Routes prefixed with `/api/v1/`

**Contracts** (Pydantic schemas outline):

| Endpoint | Method | Request | Response | Auth |
|----------|--------|---------|----------|------|
| `/auth/login` | POST | `{"username": str, "password": str}` | `{"csrf_token": str}` + Set-Cookie | No |
| `/auth/logout` | POST | — | `{"message": "ok"}` + clear cookie | Yes |
| `/auth/me` | GET | — | `{"username": str}` | Yes |
| `/barbers` | GET | — | `list[BarberOut]` | No |
| `/barbers` | POST | `BarberCreate` | `BarberOut` (201) | Yes |
| `/barbers/{id}` | GET | — | `BarberOut` | No |
| `/barbers/{id}` | PUT | `BarberUpdate` | `BarberOut` | Yes |
| `/barbers/{id}` | DELETE | — | `204` | Yes |
| `/services` | GET | — | `list[ServiceOut]` | No |
| `/services` | POST | `ServiceCreate` | `ServiceOut` (201) | Yes |
| `/services/{id}` | PUT | `ServiceUpdate` | `ServiceOut` | Yes |
| `/services/{id}` | DELETE | — | `204` | Yes |
| `/appointments` | POST | `AppointmentCreate` | `AppointmentOut` (201) | No |

```python
# Schema outlines (full definitions in app/interfaces/schemas/)
class BarberCreate(BaseModel):
    name: str = Field(min_length=1, max_length=100)
    phone: str = Field(pattern=r"^\+?[\d\s-]{7,20}$")
    price: Decimal | None = None

class BarberOut(BarberCreate):
    id: UUID

class AppointmentCreate(BaseModel):
    barber_id: UUID
    service_id: UUID
    date: date
    time: time
    client_name: str = Field(min_length=1)
    client_phone: str = Field(pattern=r"^\+?[\d\s-]{7,20}$")

class AppointmentOut(BaseModel):
    id: UUID
    date: date
    time: time
    barber_name: str
    service_name: str
    client_name: str
    client_phone: str

class TimeSlot(BaseModel):
    start: str  # "09:00"
    end: str    # "09:30"
    available: bool
```

**Scenarios**:

- GIVEN valid appointment data (barber exists, service exists, slot is free)
  WHEN `POST /api/v1/appointments` is called
  THEN response is `201` with `AppointmentOut` containing `status: "pending"`

- GIVEN a date with no availability configured for the barber
  WHEN `GET /api/v1/availability/barbers/{id}/slots?date=2026-05-20` is called
  THEN response is `200` with an empty list `[]`

- GIVEN no auth cookie
  WHEN `POST /api/v1/barbers` is called
  THEN response is `401` with `{"detail": "Not authenticated"}`

- GIVEN appointment data with a past date
  WHEN `POST /api/v1/appointments` is called
  THEN response is `422` with validation error

**Dependencies**: Domain entities (capability 4), Auth (capability 6)

## 6. Auth System

**Description**: JWT in httpOnly cookie (SameSite=Lax) + CSRF token in response body. Single admin via bcrypt.

**Acceptance Criteria**:
- `POST /api/v1/auth/login` with valid credentials returns `Set-Cookie` header (httpOnly, Secure, SameSite=Lax) + `{"csrf_token": "..."}` body
- `POST /api/v1/auth/login` with invalid credentials returns `401`
- `get_current_admin` FastAPI dependency reads JWT from `access_token` cookie, verifies signature, returns admin identity
- Protected endpoints reject requests without valid cookie with `401`
- Logout clears the cookie
- Initial admin credentials are seeded via migration or first-run script

**Contracts**:

```python
# app/interfaces/schemas/auth_schema.py
class LoginRequest(BaseModel):
    username: str
    password: str

class LoginResponse(BaseModel):
    csrf_token: str

class CurrentAdmin(BaseModel):
    username: str
```

**Scenarios**:

- GIVEN an admin exists with `username="admin"` and bcrypt-hashed password
  WHEN `POST /api/v1/auth/login` with correct credentials
  THEN response includes `Set-Cookie: access_token=<jwt>; HttpOnly; Secure; SameSite=Lax; Path=/`
  AND body contains `{"csrf_token": "<token>"}`

- GIVEN an admin exists
  WHEN `POST /api/v1/auth/login` with wrong password
  THEN response is `401` with `{"detail": "Invalid credentials"}`

- GIVEN a valid JWT cookie
  WHEN `GET /api/v1/auth/me` is called
  THEN response is `200` with `{"username": "admin"}`

- GIVEN an expired JWT cookie
  WHEN a protected endpoint is called
  THEN response is `401` with `{"detail": "Token expired"}`

**Dependencies**: Admin entity (capability 4), `passlib[bcrypt]`, `pyjwt`

## 7. Redis + n8n Integration

**Description**: Redis stores temp client data (hash + 60min TTL). On appointment creation, fire-and-forget httpx POST to n8n webhook.

**Acceptance Criteria**:
- `TempClientRepository.store(key, data)` stores in `client:booking:{key}` hash with 3600s TTL
- `TempClientRepository.get(key)` returns dict or None
- Appointment creation triggers `N8nNotifier.notify(appointment_data)` as a background task
- Webhook payload includes appointment, barber, service, client details
- Webhook failure is logged — never blocks the response
- Webhook URL is empty-string-safe (no-op if not configured)

**Contracts**:

```python
# app/infrastructure/redis/client_repo.py
class TempClientRepository:
    async def store(self, key: str, data: dict, ttl: int = 3600) -> None: ...
    async def get(self, key: str) -> dict | None: ...

# app/infrastructure/webhook/n8n_notifier.py
class N8nNotifier:
    async def notify(self, event: AppointmentCreatedEvent) -> None: ...
```

**Scenarios**:

- GIVEN valid client data `{"name": "Juan", "phone": "+5491122334455"}`
  WHEN `store("session_abc123", data)` is called
  THEN Redis hash `client:booking:session_abc123` contains the data
  AND TTL is set to 3600 seconds

- GIVEN a TTL-expired key
  WHEN `get("session_abc123")` is called
  THEN `None` is returned

- GIVEN an appointment is created and `N8N_WEBHOOK_URL` is set
  WHEN `notify(event)` is called with a valid event
  THEN httpx POST is sent to the webhook URL with JSON payload
  AND the appointment creation response is returned immediately (non-blocking)

- GIVEN `N8N_WEBHOOK_URL` is empty
  WHEN `notify(event)` is called
  THEN no HTTP request is made (no-op)

- GIVEN the webhook endpoint is unreachable
  WHEN `notify(event)` is called
  THEN the error is logged and no exception propagates to the caller

**Dependencies**: Redis 7 running (Docker), httpx

## 8. Testing

**Description**: pytest with `asyncio_mode=auto`, test files mirroring `app/` layer structure. SQLite via aiosqlite for infra/interfaces tests. Domain tests are pure Python.

**Acceptance Criteria**:
- `pytest` discovers and runs all tests with a single command
- Domain tests import entities directly — no fixtures, no DB
- Infrastructure tests use `sqlite+aiosqlite:///:memory:` engine
- `conftest.py` provides `override_get_session` fixture replacing DB with SQLite
- `conftest.py` provides `async_client` fixture with dependency overrides
- Tables created/dropped per test via `Base.metadata.create_all/drop_all`
- Test coverage includes: 2+ scenarios per endpoint, 1+ per entity validation, 1+ per use case
- Tests are deterministic (no shared state between tests)

**Scenarios**:

- GIVEN a domain entity class
  WHEN the test creates an instance with valid data
  THEN assertions on instance attributes pass without any test infrastructure

- GIVEN the SQLite test engine
  WHEN `Base.metadata.create_all` is called
  THEN all entity tables are created in `:memory:`

- GIVEN the test client with overridden DB dependency
  WHEN `GET /api/v1/barbers` is called
  THEN response is `200` with an empty list `[]`

**Dependencies**: pytest ≥8, pytest-asyncio, aiosqlite, httpx (for TestClient)
